# README 开头的几个候选版本

这个文件是内部讨论，不对外发。三版 README 首屏 + 定位语的草稿，各有侧重。

用户反馈：之前的开头"痛点不够真切、价值不够清晰"。这里每版都试图把场景讲到**具体**，把 value prop 讲到**不含黑话**。

---

## 版本 A：前后不一致型（最容易共鸣）

切入点：agent 在两次会话之间行为不一致，追源才发现是它自己改了 memory。

### 开头文案

> 上周你告诉 agent "用 Python，不要 TypeScript"。它听懂了，之后一直 Python。
>
> 今天你问同一类问题，它给你 TypeScript。你打开 `CLAUDE.md`，发现 preference 那段少了一行——就是"不要 TypeScript"那句。
>
> 谁改的？什么时候改的？为什么改？你从没 commit 过这个文件，`git blame` 查不到。
>
> 这不是 "agent 缺 memory" 的问题。你给了它 memory。问题是 **这份 memory 没人管**。

### 接定位

> 你管代码用 git——改、diff、review、commit、必要时 rollback。
>
> 你管 agent 的配置文件（`CLAUDE.md` / `AGENTS.md`）用什么？多数人的答案是"我手改，改完祈祷没出事"。相当于生产环境裸改代码还不 commit。
>
> **`forge-core`** 做的事：给 agent 的配置文件这一层补上这套流程。不替代 git——补 git 覆盖不到的那半段：**"从你的长期内容编译成 agent 真正读的那份上下文"** 这中间的 review 和回滚。

### 优点
- 场景非常具体（"上周 Python，今天 TypeScript"），画面感强
- 直接戳到 builder 普遍经历过的"agent 前后不一致"痛点
- git 类比对工程师 audience 最快能理解
- 定位清晰：不替代 git，补半段

### 缺点
- 依赖 "agent 会自己改自己的 memory" 这个前提——不是所有 runtime 都这样，可能部分读者会说 "我的 agent 不会乱改"
- git 类比已经在很多项目里用过（"git for X"），有点套路

## 版本 B：分享 / 隐私型

切入点：把 `CLAUDE.md` 分享给别人时，才发现它既当真相库又当分发物。

### 开头文案

> 你把积累半年的 `CLAUDE.md` 发给队友，对方打开，看见里面有你的真名、公司内部项目名、你老婆的生日（去年你让 agent 帮你记的）。
>
> 问题不是你不小心。
>
> 问题是：这份文件**既是你的真相库，又是对外分发物**。agent 读的和你写的是同一份——没有编辑层，也没有过滤层。等到要分享的那一刻，你才发现它根本没法分享。

### 接定位

> `forge-core` 强制分层：
>
> - 一边是你改的源文件（想怎么写怎么写，带隐私也没事）。
> - 一边是 agent 读的编译产物——由 config 挑哪几段进去，哪几段留下。
> - 中间一道审核关口，在产物生成之前让你看一眼。
>
> 不是"更好的 memory"，是把真相库和分发物拆成两件事。

### 优点
- "同一份文件既是真相又是产物"这个洞见相对独特
- 隐私/分享的场景很具体，情感力量强
- 价值主张清晰："强制分层"是一句话能讲完的事

### 缺点
- 多数个人用户并没真的把 CLAUDE.md 拿出来分享过——场景共鸣人群窄
- "真相库 vs 分发物"对没搞过 build system 的人可能还是抽象

## 版本 C：生态/趋势型

切入点：Karpathy LLM wiki 之后，社区在做加法，没人做关口。

### 开头文案

> 一年前 Karpathy 发了 LLM Wiki 的设想，之后社区里做 personal OS 的项目都在做"加东西"：加 memory、加 rule、加 skill、加 index、加 retrieval、加 sync。
>
> 没人做的是——**加东西之前让你先看一眼的那道关口**。
>
> `rulesync` 同步规则，但改了就直接推。`claude-memory-compiler` 自动抓会话进 memory，但没人审核。DSPy / BAML 编译 prompt，不管长期内容。
>
> agent 读的上下文是一个**发布产物**，不是一个记事本。凡是发布产物，就需要 review / diff / rollback。

### 接定位

> **`forge-core`** 是个小工具，做且只做那道关口：
>
> - 你的长期内容（section）在一边。
> - agent 实际读的编译产物（CLAUDE.md / AGENTS.md）在另一边。
> - 中间有 `forge diff` / `forge approve` / `forge reject`，让每次变化在发布前过一道你的手。
>
> 不跟 rulesync / claude-memory-compiler 抢地盘——把它们补成"可 review 的"就够了。

### 优点
- 对追 personal-AI 趋势的 builder 圈最有效，有行业 context
- 正面承认"别的工具做了 X，我们做 X 没做的那半"——姿态诚实
- 直接切到价值点：**关口是稀缺的那件事**

### 缺点
- 信息密度高，要求读者认识 Karpathy LLM wiki / rulesync / DSPy 等——非圈内人可能一头雾水
- 对"普通一点的"读者来说门槛高

## 我倾向 A

三版都能用，A 最稳。原因：

- 共鸣面最广：前后不一致是几乎所有 agent 用户都经历过的事。B 太窄（分享 CLAUDE.md 的人少），C 太圈内。
- 痛点最真切：A 的场景有明确因果链（观察 → 追源 → 发现无迹可循），读者会自己代入。
- git 类比虽然套路，但对目标受众（工程师）最有效、阅读成本最低。

但 A 在"给持怀疑态度的人"那段里可以吸收一部分 C 的内容（对 rulesync / claude-memory-compiler 的直接对比和定位），让生态意识也进来。

**决定**：README 主开头用 A 的两段（前后不一致 + git 类比定位），不动主体；`给持怀疑态度的人` 一节吸收一部分 C 的生态诚实定位（"它们做同步 / 抓 memory / 编 prompt，我们做它们都不做的关口"）；B 的 "真相库 vs 分发物" 观点放进 `docs/design.md` 作为设计原则部分——它是好观点但不适合 hook。

# README 开头候选

内部笔记。

## 起点

v0.1 README 一开始用的是"Python / TypeScript 前后不一致"那个痛点故事（版本 A）。dxy 反馈：**C 的路线（对标生态谈缺口）比 A 好**。但 C 第一版"AI 味太重"。

AI 味指什么：

- 排比堆砌（"加 memory、加 rule、加 skill、加 index、加 retrieval、加 sync"）
- "不是 X 而是 Y"反复用
- "没人做的是——" 用破折号装腔
- 金句型结尾（"凡是发布产物，就需要 review / diff / rollback"）
- 粗体对仗过多

下面是去 AI 味的 3 个 C 变体。现在 README 用的是 γ。

---

## C-α：带 "我" 视角的版本（最有温度）

> 去年 Karpathy 抛出 LLM Wiki 的想法之后，我看过几个类似方向的项目：`rulesync`（1k 星）做规则多工具同步；`claude-memory-compiler`（800 星）让 LLM 自动抓会话整理进 memory。工具都有用。
>
> 共性：改东西进长期内容的时候是单向管道。改了就推，抓了就写，没有中间让我看一眼的那一步。DSPy / BAML 是另一层——管 prompt 编译，不管长期内容。
>
> 我要补的就是"让我先看一眼"这一步。改完源文件，`forge diff` 告诉我产物会变什么，`forge approve` 才推，不通过就丢弃。
>
> `forge-core` 只做这件事。不做 memory、不做同步、不做 prompt 编译。

**优点**：有"我"、有时间锚点（去年）、具体数字、收尾干。
**缺点**：开头提 Karpathy 可能让不了解这条线的读者懵；略长。

---

## C-β：匀称中等版

> 最近同方向的工具不少。`rulesync`（1k 星）在 Cursor / Claude Code / Copilot 之间同步规则；`claude-memory-compiler`（800 星）让 LLM 抓会话整理进 memory。各有各用。
>
> 但都一样：改动直接进长期内容，中间没人把关。
>
> `forge-core` 只做把关这一步。改完源文件，看一眼编译产物会变什么，通过了才推给 agent。其他什么都不做。

**优点**：最短；没有人称也不冷；信息密度够。
**缺点**：有点太标准化。

---

## C-γ：最冷最直接版（当前 README 用的）

> `rulesync`（1k 星）做规则同步，`claude-memory-compiler`（800 星）做会话抓取——都是单向管道，改动直接写进长期内容。DSPy / BAML 管 prompt 编译，不管长期内容。
>
> 中间缺一道关口：改完源文件，看一眼编译产物会变什么，通过了再推给 agent。
>
> `forge-core` 就做这道关口。不做 memory、不做同步、不做 prompt 编译。

**优点**：最冷最快，信息密度最高，不绕。
**缺点**：完全没有叙事温度。

---

## 如果 γ 还是偏冷/偏 AI 味，可以往这些方向调

- **换成 α**：要更有"我"、更像 rant。
- **切 β**：要不冷不热、不带人称、更标准化。
- **改副标题语气**：README 里"给持怀疑态度的人"、"硬核验证"这些小标题仍偏技术文档味，可以换成"不跟 DSPy 比体量"、"为什么 bench 做得这么弱"这种有态度的。
- **砍掉对仗粗体**：README 全文粗体再砍一半，会更像人写的。
- **加一两处啰嗦**：人写的东西允许一点冗余。比如可以在"我要补的"后面加一句"就这么简单，不多"。

如果都不对，给我一个关键词（比如"更像 rant"、"更像说明书"、"更像推文开头"），我再试几版。
